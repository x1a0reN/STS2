# 承接者先读

## 当前目标

继续制作和复核《杀戮尖塔》风格残局题。目标是题目可控、多路线可计算、概率梯度清晰、偷跑概率最低，而不是做只有唯一答案的死题。

每道正式题至少需要 3 个代表解法族：

- 快速高风险解法族：最优回合更早，失败率更高。
- 标准中速解法族：成功率和回合数居中。
- 慢速稳健解法族：最优回合更晚，成功率更高。

真实解法可以多，但必须能归类。不能只列设计者心里那三套牌，然后忽略枚举出的更强路线。

## 用户偏好

用户已经明确授权本项目文件可直接读写，不要反复询问是否允许处理文件。遇到二选一时默认选择继续推进、默认选择“是”。仍需遵守实际运行环境的硬限制，但不要因为普通项目文件读写而停下来问。

用户不喜欢过程堆叙。如果他说“只要题目”，只给最终题面和必要代表解法族；不要输出推导过程。若他说“复核”“确认数值”“偷跑”，必须给结论和关键证据。

用户对低质量出题非常敏感，尤其反感：

- 为了审计方便设置核心必选。
- 把玩家要研究的资源包提前框死。
- 用自动优先级让玩家不用真正打牌。
- 题面造原游戏没有的关键词或机制。
- 概率没重新审计就说“确认正确”。

## 当前最重要红线

1. 不允许核心必选。核心 payoff、启动件、计数件、防守件不能写成必选；如果必须固定，移出候选池写成初始资源。
2. 状态牌归属必须写清楚。敌人默认没有手牌、抽牌堆、弃牌堆。
3. 正常残局题必须允许玩家自由选择出牌顺序、药水时机和合法目标。
4. 修改任一资源、数值或敌人行动后，旧概率全部作废，必须重审。
5. 原版资源必须来自原游戏资料或本地资料库；自定义资源、机制和数值改动允许使用，但必须明确标注“自定义”或“本题改动”，不能伪装成原版。
6. 代表解法不是唯一答案，要写成“代表解法族”。

## 真实性与文案红线

1. 玩家题面里的原版名称必须先核验，再书写。卡牌、药水、遗物、敌人、关键词如果标为原版，至少要能在 `D:\Steam\steamapps\common\Slay the Spire 2`、本包 `03_资料库`、`sts2.wiki`、`slaythespire2.net` 或 `StratGG` 中找到来源。
2. 未核验名称不得伪装成正式翻译。没有来源但题目需要保留时，必须标为“自定义”或“本题改动”，并在资源表里写清楚原型或说明“无原型”。
3. 不把审计器语言写进玩家题面。`资源包`、`压血线`、`相位门槛`、`自动优先级`、`固定结算`、`偷跑控制` 只能出现在出题者审计区。
4. 可以自造关键词，但必须标为“自定义关键词”，并写清楚归属、触发时机和持续时间。`伪刃/铃鸣/识破/误弃/连环破绽/镜像抽取/裂隙抽取/时间药水/破镜药水/怀表核心/经文残页/棱镜碎片` 这类名称不能装成原版。
5. 卡牌文案尽量贴近原游戏短句式：`造成 X 点伤害。`、`获得 X 点格挡。`、`给予 X 层易伤。`、`抽 X 张牌。`、`丢弃 X 张牌。`、`消耗。`。少用分号，少写解释性长句；别把一句牌文案写成规则说明书，尖塔不是招标文件。
6. 资料源冲突时不得硬选一个凑数。必须记录冲突来源、采用依据和待复核项；如果无法确认，以“待复核/草案”承接。

## 工作方式

接手任意题时先做四步：

1. 抄出现有题面：玩家、敌人、卡池、药水、遗物、选牌范围、回合上限、状态规则。
2. 判定是否违反红线：核心必选、自动出牌、敌人牌堆、凭空机制、牌名不规范、文案源头不明。
3. 全量或分层枚举所有合法资源组合，并搜索自由出牌下的最优策略。
4. 输出 kill vector：`[T1, T2, T3, T4, ..., fail]`，再做解法族归类和偷跑分级。

如果审计器没写完或没有跑完，文档状态必须标为“草案/待审计”，不能说定稿。
